Overview

Gin Rummy is an imperfect-information card game needing both short-horizon arithmetic (deadwood counting) and long-horizon planning (meld formation). Our roadmap has three phases: (1) train a strong RL backbone against weak opponents; (2) train it against an LLM opponent that is strategically richer but operationally far slower; (3) let the two co-evolve. This report covers Phases 1–2.

Phase 1 — the RL backbone saturates against random

We sweep eight configurations (a 3×2 grid over learning rate and entropy coefficient, plus two ablations), each trained for 2M steps with 96 parallel envs on a 64-core node and evaluated over 1000 deterministic games vs a random opponent. Every configuration lands in the 98.3%–99.6% band — statistically indistinguishable (binomial 95% CI ±0.6 pp). The shared failure mode is the lever: mean reward sits at the knock value (0.5), not the gin value (1.5) — a risk/reward call only a thinking opponent can teach. That motivates Phase 2.

Phase 2 — infrastructure for LLM-in-the-loop training

A single PPO rollout fires up to ~50k opponent queries; at 0.5–3 s per 7B call a naïve loop takes hours. We decouple inference from RL with a three-tier stack:

  • Workers (one GPU each, HuggingFace backend) load the model and register themselves in a shared-filesystem registry.
  • Master (CPU) discovers live workers, load-balances, and owns a suit-symmetry prompt cache (Gin Rummy is invariant under suit relabeling, so suit-equivalent states share one entry).
  • Client (96 env subprocesses) reaches the master through the existing Ollama-compatible API — no RL code changes.
Infra finding. Loading a 7B worker from the home NFS runs at ~11 MB/s (~28 min, blows the health-check timeout). Staging weights on scratch/BeeGFS cuts this to 27 s — a 62× speedup, mandatory when launching dozens of workers.

Play it: human vs RL web client

A zero-dependency web client (HTML/JS) lets a human play any of the four trained opponents with 3-D card animations. The same LLMAgent path the RL loop uses also drives an optional LLM opponent. Validated end-to-end in headless Chromium.

First RL-vs-LLM run

PPO warm-started from the self-play champion, 64 env subprocesses, playing through the master against a pool of Qwen2.5-7B workers (requested 32, granted 14 under the shared GPU quota — elastic). The stack ran end-to-end: 19,348 load-balanced calls at ~32 q/s; the terse prompt + 64-token cap dropped per-call latency from ~16 s to ~0.4 s. PPO sustained 39 env-steps/s and completed 40k steps (~5 rollouts) in 17.5 min.

Honest result. The fine-tuned agent retains 98.2% vs random but sits at 45.4% vs the self-play champion it started from — 40k steps (against 3M for the self-play generation) is far too short to improve. This run validates the pipeline and confirms competence is retained; demonstrating that the LLM teacher helps is the Phase-3 experiment (a much longer fine-tune).

Reproducibility

Sweeps run as SLURM arrays (sweep/*.slurm). The LLM stack is slurm/master.slurm + slurm/worker.slurm (array, scratch-staged weights) + slurm/llm_train.slurm; sweep/llmplay_one.py runs the RL-vs-LLM fine-tune. Every figure here is regenerated by paper/make_figures.py from the JSON result files, and this page by paper/make_report_html.py. The typeset 4-page version is paper/main.pdf.

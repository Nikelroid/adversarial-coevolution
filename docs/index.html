Adversarial Co-Evolution of RL and LLM Agents in Gin Rummy

We trained a small reinforcement-learning (RL) agent to play Gin Rummy, then tried to make it stronger by training it against a large language model (LLM). This is the full, honest story — what we built, what worked, what didn't, and what's next.

Gin Rummy is a card game that rewards two skills at once: quick counting (how many "deadwood" points are left in your hand) and longer-term planning (building "melds" — sets and runs). Our plan has three phases:

  1. Phase 1 — train a strong RL agent against a random opponent.
  2. Phase 2 — train it further against an LLM opponent that should play smarter.
  3. Phase 3 — let the RL agent and the LLM keep improving against each other (first result below).

This report covers Phases 1–2 in full, plus the first Phase 3 result (see the end). The headline so far: the RL agent already plays near-perfectly against simple opponents, and the 7B LLM we used turned out to be a weaker player than it, so training against the LLM did not make the agent better. We then asked whether rewarding the gin directly would help, and found it does not: the gin rate stays flat no matter how large the bonus. The open problem, a teacher actually stronger than the student plus a denser learning signal for the gin, is what the next phase attacks.

One bug we fixed first

The game library (PettingZoo) quietly reset our custom scoring (the knock/gin reward values) every time the environment was re-seeded. We found and fixed this upstream (PettingZoo PR #1312); all results below use the correct rewards (knock = 0.5, gin = 1.5).

1The RL agent crushes a random opponent

We use PPO (a standard RL algorithm) with action masking — we block illegal moves so the agent can only choose legal ones. We swept 8 settings (3 learning rates × 2 exploration values, plus 2 extra variants), trained each for 2 million steps with 96 games running in parallel, and tested each on 1000 fresh games against a random opponent.

Result: every setting wins 98.3–99.6% of games — statistically they're all tied. The interesting part is the reward: it sits at the "knock" value (0.5), not the larger "gin" value (1.5). In plain terms, the agent always takes the safe early finish and never holds out for the bigger gin bonus. That gap is the room to improve, and a random opponent simply can't teach it — which is exactly why we move to a thinking opponent in Phase 2.

2Serving the LLM fast enough to train against

One training pass asks the opponent for a move up to ~50,000 times. At a few seconds per LLM call, that's hours of waiting. So we built a small serving system with three parts:

  • Workers — one per GPU, each holding a copy of the model. On startup each worker writes its address to a shared folder so it can be found (no central database needed).
  • Master — a single CPU process that discovers the live workers, spreads requests across them, and keeps a cache of answers. The cache uses a "suit-symmetry" trick: Gin Rummy plays the same if you swap the suits, so equivalent positions share one cache entry.
  • Client — the RL training just talks to the master using the same API it already spoke, so none of the RL code changed.

We run the model with HuggingFace (not vLLM) because the installed vLLM version clashed with our Transformers version.

  env (asks for a move) --> Master (CPU): cache hit? return ;  miss --> pick a worker
                                  |
              +-------------------+-------------------+
              v                   v                   v
          GPU worker          GPU worker     ...   GPU worker     (each runs Qwen2.5-7B)

Useful finding: loading a 7B model from the home filesystem ran at ~11 MB/s and took ~28 minutes — so slow the workers timed out before they were ready. Loading from fast "scratch" storage took 27 seconds — 62× faster. This is essential when launching dozens of workers. (Also handy: which model the opponent uses is decided entirely by what a worker loads, so swapping opponents is a one-line change.)

3The opponents we tried

Self-play and a pool of past versions

Training the Phase-1 agent against a frozen copy of itself for 3M steps made it beat that copy 58–61% of the time while still crushing random (~98.7%) — a real, clean step up. We also tried training against a growing pool of past versions (AlphaZero-style), but past ~10M steps it broke: its win rate against random fell from ~98.5% to 85.8%. We kept the earlier, healthy checkpoints and use self-play as the production opponent.

LLM opponents

A tiny model (OLMoE-1B) couldn't follow the move format — most of its answers failed to parse and fell back to a random move, so it effectively played at chance and taught nothing. Qwen2.5-7B, on the other hand, played genuinely well (beat random 5/5 in our check). Its careful "think out loud" prompt was slow (~16 s/call), so for training we switched to a short "just give the move" prompt and capped the output length, cutting the time per call by about 40×.

5The big experiment: RL vs LLM (and the honest result)

We took the best self-play agent and trained it for 1.5 million steps against the Qwen2.5-7B opponent through the worker pool. The serving side delivered: time per call dropped from ~16 s to ~0.4 s, the master handled ~32 calls/second, the agent ran at ~39 steps/second, and the whole run finished in a few hours.

Honest result: it did not help. Over all 1.5M steps the reward stayed flat at the knock value (~0.49) and never moved toward gin (left plot). The fine-tuned agent still beats random (99%) and the weaker agents, but it only ties the champion it started from (43% — right plot). Why: the 7B LLM is a weaker Gin Rummy player than our self-play champion, so it can't teach an already near-perfect agent anything new. Training against a weaker teacher just holds the agent in place (and nudges it slightly off).

6Limitations

  • The teacher is weaker than the student. This is the main reason there's no improvement.
  • We used a short "just give the move" prompt for speed; we haven't tested a slower, smarter prompt or a bigger model as the teacher.
  • One training seed per run, and the "Qwen beats random 5/5" check is a small sample.
  • The prompt cache gave almost no speedup at this scale (early-game positions rarely repeat).
  • The "pool of past versions" training diverged after ~10M steps.
  • Two earlier negative results (DAgger, dense reward shaping) still need a code audit before we trust them.

7Phase 3: first result

Can reward shaping make the agent go for gin? Phase 1 showed it always knocks and never gins. We resumed the champion and trained 6 reward settings × 3 seeds (pushing the gin reward from baseline up to "gin only"), 8M steps each, then scored the gin rate with fixed rewards so the settings are comparable.

The answer is no. The gin rate sits at about 1.1 to 1.5% for every setting, the same at the baseline (1.40%) as at the most extreme incentive (1.40%), while the win rate vs random stays near 98% and the agents still beat the version they started from (57 to 60%). Shaping the final reward is not the lever. The agent rarely even attempts the long run of held cards a gin needs, so the gap is about exploration and credit assignment, not the size of the bonus.

What's next. (1) Dense, step by step shaping that rewards every point of deadwood removed, not just the final outcome. (2) A stronger teacher (Qwen3-30B): this needs two GPUs per worker and did not get scheduled under the shared quota, so it moves to the next phase. (3) Learning from bulk LLM games offline (CQL distillation).

8A gold-standard yardstick

How good are our agents, really? To find out we built a gold-standard agent: it computes the mathematically optimal way to arrange a hand into melds (exact deadwood minimization), takes the up-card only when it lowers deadwood, goes gin when possible, and knocks as soon as it is allowed. It is deterministic, fast, and plays a clean expert game. We played it against every agent we trained, 600 games each, swapping seats so neither side gets a dealer edge.

Meet the agents (also the opponents in the playable game). Lower "gold win%" means a tougher opponent for the gold standard, so the self-play champion is the strongest of the learned agents.

Two findings. First, the gold-standard agent beats our best RL agent 70 to 30, so the learned agents are strong but well short of optimal play; most of the gap is in how they form melds and choose discards, not in the knock decision. Second, the gin question finally makes sense. A gold agent that holds out for a gin gins 96% of the time against a weak (random) opponent and still wins 96%, but against the champion, who knocks early, holding for gin wins only 15%: the opponent ends the game before the gin lands. Going for gin pays off only against weak opponents. That is exactly why the RL agent, trained in self-play against strong opponents, rationally learned to knock, and why rewarding the gin directly (Phase 3) did not move it.

What this is for. We use the gold-standard agent strictly as a benchmark yardstick, never as a teacher. The project's real goal is a framework that develops strong RL using an LLM as the guide, for tasks where no clear algorithm or gold standard exists. Gin Rummy is just the testbed where a yardstick happens to exist, so we can measure how close LLM-guided RL gets. Phase 4 starts with LLM-DAgger: a language model proposes moves the RL agent learns from, then RL improves beyond it. The open question, raised in Phase 2, is whether a sub-optimal LLM can still bootstrap a strong RL policy.

9Is plain PPO the right algorithm?

We trained the same masked self-play task (from scratch vs the champion, 2M steps) with two on-policy RL algorithms, 2 seeds each, to isolate the algorithm. TRPO clearly beats PPO, consistently across both seeds and every opponent.

Algorithmwin% vs randomvs championvs gold
PPO (2 seeds)81.131.015.0
TRPO (2 seeds)91.850.522.5

Honest read. A different RL algorithm does help here: TRPO's trust-region update gives a ~50% larger win rate vs the gold standard than plain PPO. Two caveats also worth stating plainly: (1) these agents train from scratch in only 2M steps, so absolute numbers are modest (the 12M-step champion itself wins ~30% vs gold) — the valid signal is the comparison, not the absolute strength; (2) GRPO and DPO do not apply here — they are LLM-alignment methods (group-relative reward-model / preference fine-tuning) with no per-step environment interaction, so "they would do better" is false by construction. The genuinely promising next variant is RecurrentPPO (LSTM memory for the imperfect-information state).

10Does a learned state embedding help?

The observation is a large, sparse 208-d vector. We tried compressing it to a dense 20-d embedding two ways and using it (frozen) as the RL policy's input: a temporal embedding (self-supervised: states near each other in a game are similar) and an LLM-similarity embedding (an LLM judges how strategically similar two states are). Honest read so far. The temporal embedding hurts: a frozen 20-d bottleneck that is good at "are these two states similar" (val-corr 0.71) still discards detail the policy needs, so it loses to the raw sparse vector. The open question is whether the LLM-similarity embedding (pending) keeps more strategically-relevant structure, and whether a larger embedding or an un-frozen one recovers the lost information. That ablation is next.

11Closing the gap to gold: algorithm × reward × curriculum

The gold-standard agent beats every learned agent (the champion wins only ~30% vs gold). To push past that ceiling we run one big, controlled sweep over the three levers most likely to help, each as a one-factor ablation around a strong baseline (TRPO, balanced reward, a league-lite curriculum) so every result isolates a single cause — 12 cells, 30 runs (2–3 seeds each).

1 · Best algorithm

Our earlier study already showed TRPO > PPO under identical masking, so TRPO is the baseline; PPO is the comparison arm. We add the drop-in levers a terminal-only, long-horizon card game actually wants: a longer discount (γ=0.997, so credit reaches ~40 decisions back instead of decaying to 0.67), reward normalization, and a larger-batch / lower-variance-advantage TRPO cell. (NFSP and Deep CFR are multi-week rebuilds, not drop-ins; RecurrentPPO is deferred because our stateless evaluator would mis-score an LSTM.)

2 · Best reward (the optimal player almost never gins)

The mechanism insight comes straight from the gold benchmark: the optimal player gins only 0.7–1.7% of the time yet wins 70–99% — it wins by knocking early with low deadwood, not by chasing gin. So a reward that privileges gin is miscalibrated. We test four designs: R1 balanced (gin not privileged, the baseline), R0 high-gin (the current setting, run as a negative control — we expect gin-rate to stay flat, not rise), R2 knock-forward (bigger win bonus), and R4 early-knock (a small dense penalty per decision that rewards speed). Mechanism checks: gin-rate (should fall toward gold's) and mean game length (R4 should shorten hands).

3 · Best curriculum

A league-lite schedule ramps the opponent: random → a growing pool of the agent's own past selves plus pre-divergence milestones → self-play → a non-evicting strong tier (champion + the two LLM-derived models, sampled only late so a fresh agent is not crushed). We test the schedule (drop random in the tail — a forgetting test) and, most importantly, the sampling rule: recency-uniform vs PFSP (sample the opponents you currently lose to more often).

All 30 runs are queued on the cluster and tracked live in Weights & Biases (group phase6-curriculum). This table fills in automatically as cells finish — a SLURM watchdog re-aggregates and pushes the page after every completed run.

How to reproduce

Sweeps run as SLURM array jobs (sweep/*.slurm). The LLM stack is three SLURM jobs in slurm/: the master, a worker array (with model weights staged on fast scratch storage), and the trainer. Every plot on this page is regenerated by paper/make_figures.py from saved JSON results, and this page by paper/make_report_html.py.

# play the web game
python game/server.py --host 127.0.0.1 --port 8000      # open http://127.0.0.1:8000

# Phase-1 sweep
sbatch sweep/sweep.slurm

# LLM stack: master + GPU worker pool, then train against it
sbatch slurm/master.slurm
sbatch --array=0-31 --export=ALL,WORKER_PRESET=qwen2.5-7b,WORKER_MAXTOK=64 slurm/worker.slurm
sbatch slurm/llm_train.slurm

# benchmark the gold-standard agent vs every trained agent
python sweep/bench_gold.py

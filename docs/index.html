Adversarial Co-Evolution of RL and LLM Agents in Gin Rummy

We trained a small reinforcement-learning (RL) agent to play Gin Rummy, then tried to make it stronger by training it against a large language model (LLM). This page is the honest story of what worked, what didn't, and what's next.

The plan has three phases: (1) train a strong RL agent against a random opponent, (2) train it further against an LLM opponent, (3) let the RL agent and the LLM improve each other. We finished Phases 1–2. The headline: the RL agent already plays near-perfectly, and a 7B LLM turned out to be a weaker player than it — so training against the LLM did not make the agent better. The infrastructure all works; the open problem is getting a teacher that is actually stronger.

1The RL agent crushes a random opponent

We use PPO (a standard RL method) with "action masking" so it can only pick legal moves. We tried 8 settings, each trained for 2 million steps, then played 1000 games against a random opponent. Every setting wins 98.3–99.6% of games — they're all basically tied. But notice the reward sits at the "knock" value (0.5), not the bigger "gin" value (1.5): the agent always takes the safe knock and never holds out for a gin. That gap is the room to improve, and a random opponent can't teach it — you need a thinking opponent. That's the whole reason for Phase 2.

2Serving the LLM fast enough to train against

One training pass asks the opponent for a move up to ~50,000 times. At a few seconds per LLM call, that's hours. So we built a small serving system: many GPU workers each hold a copy of the model and register themselves; a CPU master finds them, spreads the work, and caches repeated questions; the RL training just talks to the master. No changes to the RL code.

  env (asks for a move) --> Master (CPU): cache hit? return ;  miss --> pick a worker
                                  |
              +-------------------+-------------------+
              v                   v                   v
          GPU worker          GPU worker     ...   GPU worker     (each runs Qwen2.5-7B)
Useful finding: loading a 7B model from the home filesystem took ~28 minutes (so slow the workers timed out). Loading from fast scratch storage took 27 seconds — 62× faster. Essential when launching dozens of workers.

4Play it yourself

There's a simple web game (no install) to play any of the trained agents, with card animations. The same code path that serves the RL training can also put an LLM in the seat.

5The big experiment: RL vs LLM (and the honest result)

We took the best self-play agent and trained it for 1.5 million steps against the Qwen2.5-7B opponent through the worker pool. The serving stack delivered: per-call time dropped from ~16 s to ~0.4 s, ~32 calls/second, and the whole run finished in a few hours.

Honest result: it did not help. Across all 1.5M steps the reward stayed flat at the knock value (~0.49) and never moved toward gin (left plot). The fine-tuned agent still beats random (99%) and the weak agents, but it's only even with the champion it started from (43% — right plot). Why: the 7B LLM is a weaker Gin Rummy player than our self-play champion, so it can't teach an already near-perfect agent anything new.

6Limitations

  • The teacher is weaker than the student. This is the main reason there's no improvement.
  • We used a short "just give the move" prompt for speed; we haven't tested a slower, smarter prompt or a bigger model as the teacher.
  • One training seed per run, and the "Qwen beats random 5/5" check is a small sample.
  • The prompt cache gave almost no speedup at this scale (early game states rarely repeat).
  • The "pool of past versions" training diverged after ~10M steps.
  • Two earlier negative results (DAgger, dense reward shaping) still need a code audit before we trust them.

7Next steps

  1. Make the teacher stronger / co-evolving (Phase 3): give the LLM example games from the RL agent's best play, or use a bigger model (e.g. Qwen2.5-32B), then alternate turns so both improve.
  2. Reward the gin directly: add a bonus for holding out for a gin / a penalty for knocking too early, since the gin decision is exactly the gap Phase 1 found.
  3. Learn from bulk LLM games offline (distillation with CQL), avoiding live LLM latency.
  4. Tighten the science: 3 seeds, and a direct "smart prompt vs fast prompt" comparison.

How to reproduce

Sweeps run as SLURM array jobs (sweep/*.slurm); the LLM stack is three SLURM jobs in slurm/ (master, a worker array with weights staged on scratch, and the trainer).
